3. Normalization Analysis

First Normal Form (1NF)

Definition: All attributes are atomic (single-valued), there are no repeating groups, and every table has a primary key.

Assessment — all four tables satisfy 1NF:

  • Every column in every table holds exactly one indivisible value per row (e.g., packedLevel holds one ENUM value, not a set).
  • There are no arrays, multi-valued columns, or repeating groups anywhere in the schema.
  • Each table has a clearly defined primary key: userId, lotId, reportId, and the composite (userId, lotId) on favorite.
  • ENUM columns (packedLevel, paymentType) store a single value from a fixed domain — this is atomic by definition.

Second Normal Form (2NF)

Definition: Must be in 1NF, and every non-key attribute must be fully functionally dependent on the entire primary key (no partial dependencies).

Assessment — all four tables satisfy 2NF:

  • user, lot, spot_report have single-column surrogate primary keys. Partial dependency is impossible with a single-column PK — 2NF is trivially satisfied.
  • favorite has a composite PK (userId, lotId). Its only non-key attribute is createdAt, which records when that specific user favorited that specific lot. It depends on the full composite key, not on just userId or just lotId alone. No partial dependency exists.

Third Normal Form (3NF)

Definition: Must be in 2NF, and no non-key attribute should be transitively dependent on the primary key through another non-key attribute.

Assessment — all four tables satisfy 3NF:

  • user: firstName, lastName, username, and password each depend directly on userId. None determines another. No transitive dependency.
  • lot: name, address, totalCapacity, paymentType, and coordinates all describe the lot identified by lotId. No non-key attribute determines another (e.g., address does not determine capacity). No transitive dependency.
  • spot_report: packedLevel, hasOpenSpots, reportedAt, and notes all describe a single reporting event keyed by reportId. The foreign keys userId and lotId are relationship attributes — they are not functional dependencies from non-key attributes. No transitive dependency.
  • favorite: createdAt depends on the composite PK (userId, lotId). There are no other non-key attributes that could form a transitive chain. No transitive dependency.

Boyce-Codd Normal Form (BCNF)

Definition: For every non-trivial functional dependency X → Y, X must be a superkey.

Assessment — all four tables satisfy BCNF:

  • In user, lot, and spot_report, the only non-trivial functional dependencies are from the surrogate PK to all other attributes. The PK is a superkey by definition. No violation.
  • In favorite, the composite PK (userId, lotId) → createdAt is the only non-trivial FD. (userId, lotId) is a superkey. No violation.
  • username in user is UNIQUE, meaning username → userId, firstName, lastName, password also holds. Since username is a candidate key (and thus a superkey), this does not violate BCNF.

All four tables are in BCNF.

4. Design Decisions

  • Surrogate keys (AUTO_INCREMENT INT) on user, lot, spot_report. Natural keys (e.g., username, lot name) can change over time. Surrogate keys provide stable references across the lifetime of the application and are more efficient as join keys on the high-volume spot_report table.
  • ENUM for packedLevel and paymentType. These fields have a fixed, small domain known at schema design time. Using MySQL ENUM enforces domain integrity at the database layer — invalid values are rejected before reaching the application — and is more storage-efficient than VARCHAR. The 5-level congestion scale (Empty → Full) also maps naturally to an ordered ENUM for CASE-based sort expressions used in analytics queries.
  • NULL-able notes and GPS coordinates. spot_report.notes is genuinely optional — not every reporter has a useful observation. Storing NULL (rather than an empty string) is semantically correct and avoids polluting text-search queries. Similarly, lot.latitude/longitude may not be known for every lot; NULL is the appropriate representation of missing geospatial data.
  • Composite PK (userId, lotId) on favorite. The natural key already uniquely identifies the relationship (a user can favorite a lot at most once). Using it as the PK avoids adding an unnecessary surrogate key, self-documents the uniqueness constraint, and eliminates the need for a separate UNIQUE index.
  • ON DELETE CASCADE on favorite FKs. If a user or lot is removed, their associated favorites become meaningless. Cascading the delete maintains referential integrity automatically without requiring application-level cleanup.
  • Indexes idx_report_lot and idx_report_time. The spot_report table is the highest-volume table (1200+ rows, growing continuously). The two most common access patterns are filtering by lotId (report history page: WHERE sr.lotId = ?) and filtering/grouping by reportedAt (analytics: WHERE packedLevel IN (...) with GROUP BY HOUR(reportedAt)). Without these indexes, both queries would require full table scans. With them, MySQL uses the B-tree index for efficient range scans.
  • reportedAt DATETIME DEFAULT CURRENT_TIMESTAMP. Setting the timestamp at the database layer (rather than in application code) ensures consistency regardless of server time zones, clock drift, or application bugs. It also means the INSERT statement does not need to supply this column, simplifying the prepared statement.
